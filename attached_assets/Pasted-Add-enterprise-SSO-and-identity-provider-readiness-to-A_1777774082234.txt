Add enterprise SSO and identity-provider readiness to ATMEMLY.

Do not rebuild.
Keep existing Drizzle/Express backend, existing frontend, and current database.

Goal:
Support enterprise-grade user identity management similar to Keycloak, while keeping the current local login system working.

Important branding rule:
- Replace all Khidma references with ATMEMLY
- JWT/session/system messages must say ATMEMLY where user-facing
- Do not introduce any new Khidma references

Required SSO providers:
1. Keycloak / OpenID Connect
2. Google OAuth
3. LinkedIn OAuth
4. Microsoft Entra ID / Azure AD ready
5. Generic OIDC provider configuration

Core requirements:
- Keep existing email/password login
- Add SSO login as optional
- Support account linking between local account and SSO identity
- Support enterprise tenant-based login later
- Support role mapping from identity provider claims
- Support admin-controlled SSO configuration
- Support forced SSO for selected organizations later

Database additions using Drizzle migrations:

- login_audit_logs if not already covered

- GET /api/admin/sso/audit

SSO logic:
1. User clicks SSO login
2. Backend redirects to provider authorization URL
3. Provider returns authorization code
4. Backend exchanges code for tokens
5. Backend validates ID token signature and issuer
6. Backend reads user claims
7. Backend finds linked user identity
8. If found, login user and issue ATMEMLY JWT tokens
9. If not found and autoProvisionUsers is enabled, create user
10. If not found and autoProvisionUsers is disabled, show account linking or access denied
11. Store login audit event
12. Apply role mapping only if configured and allowed

Account linking rules:
- If SSO email matches existing verified user, allow secure linking
- Require logged-in user confirmation before linking where possible
- Prevent linking one SSO identity to multiple users
- Allow user/admin to unlink provider unless SSO is enforced

Admin can:

- Configure allowed email domains

- Configure role mapping
- Test provider connection
- View SSO login audit logs
- Set default login method
- Enable/disable local password login globally or per organization later

Frontend requirements:

Security requirements:
- Validate OIDC state and nonce
- Prevent CSRF in SSO callback
- Validate issuer, audience, expiry, and signature
- Use PKCE where supported
- Never expose client secrets to frontend
- Store provider secrets securely
- Add audit logs for every SSO login, failed login, provider config update, and account link/unlink
- Rate-limit SSO start/callback endpoints
- Prevent role escalation from untrusted claims
- Only SUPER_ADMIN can configure SSO providers

Keycloak-specific:
- Support issuer URL such as:
  https://keycloak.example.com/realms/{realm}
- Auto-discover OIDC metadata from:
  /.well-known/openid-configuration
- Support role claim mapping from:
  realm_access.roles
  resource_access.{clientId}.roles
  groups
- Map Keycloak roles to ATMEMLY roles only through admin-approved mapping

Microsoft Entra ID readiness:

- Support tenant issuer URL
- Support email/domain-based organization mapping later

Final goal:
